Mental model: from source code to executable

source code (.c)
      ↓
compiler (gcc)
      ↓
executable (binary file)
      ↓
run on LS6

Key idea:

  • source code is human-readable
  • executables are what the system runs

Why compilation matters in HPC

On HPC systems:

  • software is often not globally installed
  • compiler versions matter
  • toolchains affect behavior
  • performance depends on how code is built

Two compilers you will hear about

  • gcc → C
  • g++ → C++

Today we focus on C with gcc.

Modules first, then compilation

Before compiling, load a compiler environment:

module avail
module load gcc
module list

Key idea:

  • on clusters, software is often environment-based

Check what compiler you are actually using

which gcc
gcc --version

Why this matters:

  • confirms the command exists
  • confirms which version is active
  • avoids guessing

Your first C program

Create a file:

nano hello.c

Add:

#include <stdio.h>

int main() {
    printf("Hello from C on LS6\n");
    return 0;
}

Compile the program

gcc hello.c -o hello

What that means:

  • hello.c = source file
  • -o hello = name of the executable

Run the executable

./hello

Expected result:

Hello from C on LS6

Inspect what you created

ls -l

You should now see at least:

  • hello.c
  • hello

Key idea:

  • source file and executable are different files

Mental model: what just happened?

You:

  • wrote source code
  • used a compiler
  • created an executable
  • ran the executable from the shell

That is the core compiled workflow.

Add a common optimization flag

gcc hello.c -o hello -O2

Useful levels:

  • -O0 = no optimization
  • -O2 = standard choice
  • -O3 = more aggressive

For beginners, -O2 is a good default.

Very small C++ contrast

g++ hello.cpp -o hello_cpp

Key idea:

  • same overall workflow
  • different compiler for C++

We are not going deep into C++ today.

Automating builds with a Makefile

Create a file named Makefile:

all:
    gcc hello.c -o hello

Then run:

make

Why this matters:

  • avoids retyping build commands
  • scales to larger projects

Compiling on login nodes, running on compute nodes

Good rule of thumb:

  • compile on login nodes for small builds
  • run heavier workloads through SLURM

That keeps the workflow aligned with the earlier workshops.

Running compiled code through SLURM

Example run.slurm:

#!/bin/bash
#SBATCH -J compile_test
#SBATCH -N 1
#SBATCH -n 1
#SBATCH -t 00:05:00
#SBATCH -p normal

./hello

Submit with:

sbatch run.slurm

Common beginner failures

  • forgot module load gcc
  • typo in source code
  • forgot -o hello
  • tried to run from the wrong directory
  • executable is missing because compile failed

Debug checklist for compilation

Check in this order:

module list
which gcc
ls -l
gcc hello.c -o hello
./hello

If compilation fails, read the compiler message carefully.
